# 19 — Spreadsheet Import Contract

## 1. Classification — RESOLVED (DEC-37)

**The financial workbooks are NOT required to implement, verify, or ship the system.**

| Question | Answer |
|---|---|
| Required for product **behavior**? | **No.** No requirement, endpoint, calculation, constraint or radar path depends on them. |
| Required for **correctness verification**? | **No.** The calculation engine is verified by synthetic fixtures (§4) with hand-computed expected values — a stronger oracle than a spreadsheet whose own arithmetic was found broken. |
| What do they actually affect? | **Only the one-time historical data load** (Phase 6) and the column-name binding inside the loader. |

**They improve mapping fidelity. They do not gate the build.**

There is a further reason not to treat them as an oracle: the source analysis found the workbooks
contain **typed-over cash formulas**, 15 spellings for ~8 fee types, accented/unaccented status
duplicates, an invisible partner, and 4 orphaned clients. A file with known-wrong arithmetic cannot
validate a calculation engine. Synthetic fixtures with known-correct values do.

**Codex must never pause for these files.** Phase 6 is built and tested against synthetic workbooks
generated by the plan itself. Swapping in the real files is a data operation (§5), not development.

## 4. Normalization, validation and reconciliation

**Header matching** reuses the canonical scraper's `chave_texto()`: NFD-decompose, strip combining
marks, collapse whitespace, casefold. So `Descrição`, `DESCRICAO` and `descrição ` all match
`descricao`. Exact match first, then substring — the same two-pass strategy as `localizar_coluna()`.

**Closed-vocabulary mapping** (the loader's core job). Every unmapped value is a **hard error with
its source cell**, never a silent default:

```
status:  "aguardando êxito" | "aguardando exito" | "aguard. exito"  -> aguardando_exito
         "sem êxito" | "sem exito"                                  -> sem_exito
         "ativo" | "em andamento"                                   -> ativo
         "encerrado" | "finalizado"                                 -> encerrado
         "proposta" | "pendente" | "em negociacao"                  -> proposta

tipo_honorario: 15 observed spellings -> 8 enum values
         "fixo unico" | "fixo único" | "unico"                      -> fixo_unico
         "fixo mensal" | "mensal"                                   -> fixo_mensal
         "fixo parcelado" | "parcelado"                             -> fixo_parcelado
         "exito puro" | "êxito puro" | "so exito"                   -> exito_puro
         "sucumbencia" | "sucumbência"                              -> sucumbencia
         "fixo + exito" | "fixo e exito" | "fixo+exito"             -> fixo_exito
         "exito + sucumbencia" | "exito e sucumbencia"              -> exito_sucumbencia
         "fixo + exito + sucumbencia"                               -> fixo_exito_sucumbencia
```

**Validation behavior, by severity:**

| Condition | Behavior |
|---|---|
| Unmapped enum value | **ERROR** — row rejected, reported with sheet/row/column/value |
| Missing required column | **ERROR** — abort that sheet, report available headers |
| Partner name not in `parceiros` | **AUTO-CREATE** + flag for review (fixes the invisible-partner defect) |
| `MENSAIS` row with no matching contract | **AUTO-CREATE** a stub contract, `status='proposta'`, `revisar=true` + flag (fixes the 4 orphaned clients) |
| Duplicate fixed cost (same description) | **WARN** — both loaded, surfaced as suspected duplicate |
| Typed cash balance ≠ recomputed | **WARN** with the per-month delta |
| Value > 100 in a percent column | **ERROR** — never clamped |
| `recebido = true` with no receipt month | **WARN** — falls back to `mes_esperado`, flagged |

**Idempotency:** each source row yields a deterministic natural key
(`sha256(sheet | row-identity)`) recorded in `import_log`. Re-running skips rows already imported
with an unchanged hash, updates changed ones, and never duplicates. Every decision (mapping applied,
entity auto-created, warning raised) is written to `import_log` with sheet, row, column, original
value and resulting value — an auditable record, per the source requirement.

**Order:** `parceiros` → `contratos` → `parcelas` → `custos_fixos` → `parametros` → `lancamentos` →
full recalculation of every month from scratch.

## 5. Synthetic fixtures — how Phase 6 is built and tested TODAY

`tests/fixtures/planilhas/gerar_sinteticas.py` generates both workbooks with openpyxl, reproducing
**every** structural defect on purpose:

| Fixture | Encodes |
|---|---|
| `fluxo_caixa_sintetico.xlsx` | CONFIG + 12 month sheets; 3 months with hand-typed balances contradicting the entries; duplicate "Contador" cost |
| `contratos_sintetico.xlsx` | CONTRATOS/ENCERRADOS/PENDENTES/MENSAIS; both `Aguardando êxito` and `Aguardando exito`; 15 fee-type spellings; a partner absent from any contract sheet; **4 `MENSAIS` clients with no contract row** |
| `esperado.json` | Hand-computed expected end state: row counts, enum distribution, recomputed monthly chain, DRE, balance, receivable/overdue split |

Because the expected values are **derived by hand from the fixture inputs**, they are a correct
oracle — unlike the real workbooks, whose arithmetic is known to be broken.

TEST-LOAD-01…05 run entirely against these. **Phase 6 reaches "done" with no external file.**

## 6. Replacement procedure — when the real workbooks arrive

Data operation, not development. No code change unless step 2 fails.

1. Place both files under `data/import/` (git-ignored — they contain client financial data).
2. `python -m app.loader --validate-only --fluxo <A> --contratos <B>`
   → prints resolved header→column bindings, unmapped values, and every warning. **No writes.**
3. If any header is unbound, add the alias to `COLUMN_ALIASES` in the loader — a **data-mapping
   table edit**, deliberately isolated from logic so this step never becomes a code change.
4. Re-run `--validate-only` until clean.
5. `python -m app.loader --commit` → transactional; full recalculation on success.
6. Review `import_log` for auto-created stub contracts and duplicate-cost warnings.
7. Compare the reported typed-vs-recomputed cash deltas against expectation, then archive the report.

**Rollback:** the load runs in one transaction against a fresh database. Failure leaves nothing
behind; re-run after fixing.
